# Qualitative Comparison: AEA vs. Adaptive Retrieval Systems

> **Purpose.** This table addresses a reviewer concern about missing baseline comparisons. It
> positions Adaptive External Attention (AEA) against five established adaptive retrieval systems
> along dimensions that capture architectural and design-level differences. Because the systems
> differ substantially in task scope, retrieval modality, and evaluation protocol, this is an
> *architectural* comparison rather than a head-to-head benchmark; direct metric comparison would
> not be methodologically sound (see positioning paragraph below).

## Positioning Paragraph

### What AEA shares with each system

**FLARE** and AEA share the core insight that retrieval is not always necessary and that triggering retrieval only when evidence is insufficient is more efficient than single-pass or fixed-schedule approaches. Both use inference-time heuristics rather than trained routing components, making them directly deployable without annotated data. AEA's coverage check is philosophically similar to FLARE's confidence-threshold check — both are proxy signals for "do I have enough to continue?"

**Self-RAG** and AEA both treat avoidance as a first-class action. Self-RAG's `[Retrieve]` token can suppress retrieval entirely; AEA's Condition 1 can terminate the pipeline after a single operation. Both systems also maintain some notion of self-assessment: Self-RAG uses reflection tokens to evaluate whether retrieved passages are relevant or supportive; AEA uses relevance scoring within the workspace to assess coverage. Both explicitly optimise for answer quality rather than retrieval completeness.

**Adaptive-RAG** and AEA both perform pre- or mid-pipeline routing to select the appropriate retrieval depth. The three-tier complexity classification in Adaptive-RAG (none / single / multi-step) parallels AEA's three-condition decision procedure (stop / entity escalation / lexical fallback). Both systems identify a "do nothing extra" path as the most efficient outcome for simple queries.

**IRCoT** and AEA are both iterative systems that use intermediate reasoning state to guide subsequent retrieval. IRCoT's use of the most recent CoT sentence as a retrieval query is an instance of the same principle underlying AEA's discovery-state tracking: what you have already found should determine what you look for next. Both improve substantially over one-shot retrieval on multi-hop tasks.

**CRAG** and AEA both include an explicit evaluation step that assesss whether retrieved evidence is of sufficient quality before committing to generation. CRAG's three-way classification (Correct / Incorrect / Ambiguous) is structurally analogous to AEA's three-condition routing check. Both systems also recognise that corrective action — querying a second source when the first is insufficient — is valuable, and both maintain a fallback path (CRAG to web search; AEA to BM25 or entity graph).

### What AEA does differently

AEA differs from all five systems along three axes that are not present in combination elsewhere.

**Heterogeneous address spaces.** FLARE, Self-RAG, IRCoT, and CRAG each operate over a single retrieval modality or at most two (CRAG adds web search as a fallback). Adaptive-RAG routes across pipeline configurations that differ in depth (single-hop vs. multi-hop) but remain within the same substrate family. AEA routes across three qualitatively distinct address-space types — dense semantic, BM25 lexical, and entity co-occurrence graph — whose output distributions, recall profiles, and cost characteristics differ substantially. The entity graph substrate, in particular, has no analogue in any of the five comparison systems: it is designed for relational chain traversal rather than ranked document retrieval.

**Explicit cost accounting in the evaluation objective.** No baseline system builds retrieval cost directly into its primary evaluation metric. FLARE, Self-RAG, Adaptive-RAG, and IRCoT report standard QA accuracy or EM/F1; CRAG reports accuracy improvements relative to a RAG baseline. AEA's primary metric, Utility@Budget, explicitly penalises unnecessary operations (μ=0.30) and rewards recall-precision jointly (η=0.50). This means AEA's results cannot be directly compared to existing benchmarks — a system that retrieves exhaustively and scores perfectly on recall would score near zero under Utility@Budget.

**Dual-state representation with workspace management.** AEA is the only system that maintains a structured two-component state — discovery state (what has been located) and knowledge state (what has been verified) — and an explicit bounded buffer with pinning and eviction. Self-RAG's state is implicit in the generation stream; IRCoT's state is the accumulated CoT; the other systems maintain no persistent state between retrieval steps. This distinction allows AEA to differentiate "I looked and found nothing" from "I have not yet looked," avoiding redundant exploration.

**Avoidance as the dominant contributor.** The ablation study (Table 4) shows that removing selective avoidance (`abl_always_hop`) is catastrophic (Δ = −0.1146 on HotpotQA), while the specific substrate chosen for escalation is largely inert (`abl_no_entity_hop` is net-positive). This empirical finding — that the value of the policy comes from not retrieving rather than from choosing the right substrate — would be invisible to any system that does not include the null action in its routing vocabulary.

### Why the comparison is not apples-to-apples

Five sources of incomparability make direct metric comparison between AEA and these baselines methodologically unsound, and reviewers should treat the table above as *architectural positioning* rather than empirical ranking.

1. **Task scope.** FLARE, Self-RAG, and CRAG are evaluated on downstream QA accuracy after full generation by a language model. AEA reports retrieval-only metrics (Support Recall, Support Precision, Utility@Budget) without downstream generation. A fair comparison would require running all systems on the same benchmark with the same generation model and scoring protocol.

2. **Benchmark design.** FLARE and IRCoT use standard open-domain multi-hop QA benchmarks (HotpotQA full split, 2WikiMultihopQA). AEA uses a 100-question bridge subset of HotpotQA and a synthetic heterogeneous benchmark (Heterogeneous v2) constructed specifically to stress-test multi-substrate retrieval. Baseline systems were not designed for or evaluated on Heterogeneous v2.

3. **Evaluation metric.** Utility@Budget is a novel composite metric not used by any comparison system. Converting their reported accuracy/EM/F1 into U@B is not possible without access to per-question operation traces.

4. **Training assumptions.** Self-RAG and Adaptive-RAG require labelled training data and fine-tuning. AEA and FLARE require none. Comparisons that conflate trained and heuristic systems risk attributing performance differences to architectural choices when they may reflect the size of the training set.

5. **Retrieval oracle access.** IRCoT and FLARE assume access to a Wikipedia corpus via BM25 or dense retrieval at scale. AEA's experiments operate over a closed distractor set (10 paragraphs per question in HotpotQA Bridge) and a constructed synthetic corpus. Recall numbers are not comparable across these settings.

A rigorous apples-to-apples comparison would require: (a) a shared benchmark covering all task types, (b) a shared retrieval corpus and implementation of each substrate, (c) a shared generation backbone to convert retrieved passages into answers, and (d) reporting both accuracy and cost under a unified metric. We identify this as an important direction for future work.
